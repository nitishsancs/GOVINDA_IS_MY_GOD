ROOT CAUSE ANALYSIS
================================================================================

[CRITICAL] F1 — UNBOUNDED USER CONTEXT INJECTION
  The UserMemoryManager accumulates interaction history.
  This context is injected into EVERY retrieval query via:
    effective_query = f"{query_text}\n\n[User Context]: {user_context}"
  By Q11, user_context contains summaries of 10 previous Q&A pairs,
  causing retrieval input tokens to explode from 10K to 190K+.
  This happens because the LLM classifier/locator receives the
  entire bloated query for EVERY sub-query retrieval in the Planner.
  FIX: Cap user_context to 2000 tokens or last 3 interactions.

[HIGH] F2 — SYNTHESIS DOMINATES PIPELINE TIME
  Synthesis is 59% of server time on average.
  The Planner runs N sub-query retrievals in parallel, then
  synthesizes from all merged sections. Each sub-query retrieval
  includes the bloated user_context.
  FIX: Cache synthesis for near-duplicate queries within a theme.
       Reduce sub-queries to max 4.

[MEDIUM] F3 — QUERY INTELLIGENCE IS STALE
  QI suggested the same 10 nodes for ALL 30 queries.
  QI avoided the same 1 node for ALL queries.
  The learning loop records facts but the suggestions don't adapt
  because they're based on embedding similarity to the query,
  and all ALM queries have similar embeddings.
  FIX: Weight QI suggestions by recent key_term overlap, not just
       embedding similarity.

[MEDIUM] F4 — ALL QUERIES CLASSIFIED AS MULTI_HOP
  Even simple 'explain' and 'why' variations get the expensive
  multi-hop Planner path with parallel sub-query retrieval.
  FIX: After the first multi_hop query in a theme achieves
       high precision, classify subsequent variations as single_hop.

[LOW] F5 — RAPTOR INDEX NEVER FIRES
  The 'Know What Matters' loop contributed 0 candidates.
  This entire learning subsystem is inactive during normal queries.
  FIX: Trigger raptor.build_index after N queries on a doc.

[INFO] F6 — RETRIEVAL TIME IS CONSTANT
  Retrieval time: 40s to 98s (avg 58s)
  The memory system is NOT reducing retrieval time.
  Retrieval does the same 6 LLM calls every time.
